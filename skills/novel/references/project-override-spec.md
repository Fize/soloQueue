# 项目红线外挂协议（通用）

> 本文件定义技能如何**自动发现、加载、应用**项目特化的红线规则。
> 技能本体**不内置**任何与具体项目相关的规则。所有项目特化内容（人物 / 设定 / 体系 / 地名红线，以及项目特有的身份言行规范）必须通过本协议注入。

---

## 一、为什么需要项目红线外挂

`novel` 技能是**通用网络小说创作技能**，可以被复制到任意新项目使用。但每个具体的小说都有自己**独一无二**的设定红线，例如：

- 某个角色的身世红线（不能写成什么）
- 某类物品的性质红线（不能被当作什么使用）
- 某个职业 / 体系的独立性红线（不能被错误关联）
- 项目特有的身份言行规范（体系名、称谓、禁忌）

这些规则**不可能**沉淀进通用技能，否则技能就不再通用。因此技能通过"外挂协议"在运行时**从项目侧**读取这些规则，强制合入**质量质检分支 (quality-editor)** 的检查序列。

---

## 二、发现优先级（自动搜索）

技能进入任意工作流时，**必须**在当前项目目录下按以下顺序搜索项目红线文件，**找到第一份即载入**：

```
优先级 1: <project-root>/novel-project.md
优先级 2: <project-root>/项目红线.md
优先级 3: <project-root>/CODEBUDDY.md
优先级 4: <project-root>/CLAUDE.md
```

### 2.1 就近原则

若当前项目存在子目录（如 `<workspace>/tianyuanjian/`），且子目录下也有红线文件，**优先使用子目录红线文件**。

示例发现顺序：

```
1. <workspace>/tianyuanjian/novel-project.md  ← 找到就停
2. <workspace>/tianyuanjian/项目红线.md
3. <workspace>/tianyuanjian/CODEBUDDY.md
4. <workspace>/novel-project.md
5. <workspace>/项目红线.md
6. <workspace>/CODEBUDDY.md
7. <workspace>/CLAUDE.md
```

### 2.2 多文件处理

如果项目同时存在多份红线文件，**只载入优先级最高的一份**，不做合并。技能应在首次回复中告知用户"发现多份红线文件，已载入 X，忽略 Y、Z"。

## 三、项目红线文件推荐结构

> 项目侧创建红线文件时，**推荐**遵循本结构以获得最好的加载效果。技能会按 Section 标题识别内容。

```markdown
# {项目名} · 项目红线

> 本文件是 {项目名} 的项目特化规则集，由 novel 技能自动发现并强制执行。

## 项目元信息

- **项目根目录**：<workspace>/tianyuanjian/  ← 告诉技能以哪里为根
- **题材**：修真 / 玄幻 / 都市 / 科幻 / ...  ← 影响默认身份言行规范加载
- **已完成卷数**：1
- **当前进度**：卷 1 · 第 12 章

## 体系文档清单

> 列出项目已有的体系文档，技能在"章节创作前置"时按需读取。

- 核心世界观总纲.md
- 修炼体系.md
- 功法体系.md
- 人物体系.md
- ...

## 禁用词追加

> 追加到通用 12 条禁用词之外的项目特化禁用词。

- 某个项目专属的 AI 风味词
- ...

## 项目红线（Must）

> 每条红线一小节，描述"不得写成什么"、"违反即 FAIL"。

### 红线 · {名称}

**规则**：<具体规则描述>

**违规示例**：

- ...

**合规示例**：

- ...

**检查方式**：<告诉质量质检分支 quality-editor 怎么检查>

### 红线 · ...

## 身份言行特化（覆盖或补充 anti-ai-patterns §六）

> 若项目题材需要覆盖默认的修玄身份言行规范，在此声明。

- 称谓规范：
- 场景规范：
- 禁忌用语：

## 路径覆盖（可选）

> 若项目使用与 output-spec 模板不同的路径，在此声明。

- 正文：<覆盖路径>
- 蓝图：<覆盖路径>
- 里程碑：<覆盖路径>

## 工作流入口提示（可选）

> 告诉用户"说什么话"可以触发什么工作流，与 command-routing 基础映射叠加。
```

## 四、加载协议（技能执行步骤）

当技能找到项目红线文件后，按以下步骤处理：

### Step 1 · 元信息解析

从"项目元信息"节读取：

- **项目根目录**：作为所有路径计算的基准
- **题材**：决定是否加载 `writing-conventions.md` 的修玄默认规范
- **进度信息**：辅助自动路由（例如"写下一章"定位最新章节+1）

### Step 2 · 体系文档清单抽取

从"体系文档清单"节抽取文件名列表。这些文件在"章节创作前置"时按需读取（见 [output-spec.md](output-spec.md) §4.1 第 6 步）。

### Step 3 · 禁用词叠加

把"禁用词追加"节的词汇，**叠加**到 [quality-checklist.md](quality-checklist.md) R1 的 12 条通用禁用词上。通用禁用词不可被删除，只能被追加。

### Step 4 · 项目红线注入

把"项目红线"节的每一条，作为**独立红线项**注入质量质检分支 `quality-editor` 的检查序列。每条红线需要：

- 一个唯一标识（用名称即可）
- 一个规则描述（quality-editor 据此判断是否违反）
- 一个检查方式建议（可选，用于指导 quality-editor 如何具体检测）

注入后的检查时序：先跑通用红线 R1-R9，再跑触发标记专项 T1-T4，最后跑项目红线。**任一失败即回修。**

### Step 5 · 身份言行覆盖

若项目红线文件声明了"身份言行特化"，则：

- 覆盖 [anti-ai-patterns.md](anti-ai-patterns.md) §六的默认修玄规范
- 覆盖 [writing-conventions.md](writing-conventions.md) §三的修玄身份规范

若未声明，则保持默认修玄基线。

### Step 6 · 路径覆盖

若项目红线文件声明了"路径覆盖"，则覆盖 [output-spec.md](output-spec.md) §二的对应路径模板。

## 五、降级行为（找不到红线文件）

### 5.1 降级策略

若搜索完优先级 1-4 的所有位置都没找到红线文件，技能**不报错、不拒绝工作**：

- 仅按技能 references 的通用规则运行
- 在首次回复中**明确提示**用户：

> ⚠️ 未发现项目红线文件。技能将按通用规则运行，但强烈建议在项目根创建 `novel-project.md`，以注入项目特化的红线规则（角色设定、体系、身份言行等）。
>
> 参考模板见 [project-override-spec.md](project-override-spec.md) §三。

### 5.2 部分降级

若找到红线文件但**缺少推荐结构**的某些节（如没有"体系文档清单"节），技能：

- 已有的节按协议加载
- 缺失的节按默认值运行（如未声明题材 → 默认修玄；未声明体系清单 → 自动扫描项目根目录的 `*.md` 文件）
- 在回复中给出建议："建议在 `novel-project.md` 中补充 XXX 节以获得更好的检查效果"

## 六、与通用规则的关系

项目红线与技能通用规则的关系是**叠加**，不是**替换**：

| 场景 | 行为 |
|---|---|
| 项目追加禁用词 | 与通用 12 条叠加，扩展禁用词表 |
| 项目新增红线项 | 追加到 R9 之后，作为独立红线项 |
| 项目声明身份言行特化 | **覆盖**默认修玄基线 |
| 项目声明路径覆盖 | **覆盖**默认路径模板 |
| 项目修改段落上限 | **不被允许**（通用硬红线） |
| 项目禁用凤头 / 豹尾要求 | **不被允许**（通用硬结构） |

### 6.1 不可覆盖的通用红线

以下**任何项目**都不得通过红线文件删除或弱化：

- 段落 ≤ 8 行
- 凤头不得环境描写开局
- 豹尾必须悬念断章
- 关键场景五感 ≥ 3
- 设定解说 ≤ 10%
- 每章爽点 ≥ 2
- 五章节点里程碑必须含 A + B 两节

项目红线只能在这些之上**追加**更严格的约束，不能放宽。

## 七、常见用法示例

### 示例 A · 追加禁用词

项目红线文件中：

```markdown
## 禁用词追加

- 本项目中的"现代爽文专用语"：系统提示、恭喜宿主、任务完成
```

效果：质量质检分支 `quality-editor` 在 R1 红线扫描时，除了通用 12 条，还会扫"系统提示"、"恭喜宿主"、"任务完成"。

### 示例 B · 人物设定红线

项目红线文件中：

```markdown
## 项目红线（Must）

### 红线 · 某圣物的性质

**规则**：某圣物只是家族信物，不是"记录秘密"的载体。严禁写成账本、记录、或"知道太多秘密"的来源。

**违规示例**：

- 他翻开圣物，里面记着三十年来所有见不得光的交易。

**合规示例**：

- 他把圣物握在掌心，它温热如故，什么也没告诉他。

**检查方式**：扫描正文是否把该圣物描述为"记载"、"账本"、"记录"、"知晓秘密"的容器。
```

效果：此红线作为独立项注入质量质检分支 `quality-editor`，任何把该圣物写成"信息载体"的段落都会 FAIL。

### 示例 C · 体系解耦红线

项目红线文件中：

```markdown
### 红线 · 职业等级与修炼境界解耦

**规则**：炼丹 / 炼器 / 阵法 / 风水 / 符箓 等职业等级与修炼境界**不是硬性对应关系**。高天赋者可借灵石、辅助法宝等方式，在较低修为下达到较高职业等级。

**违规示例**：

- 他只是筑基期修为，所以顶多只能炼制筑基期丹药。

**合规示例**：

- 他虽只是筑基期修为，但借助双羽灵石与本命丹炉的加持，炼出了结晶期才能成丹的引丹。

**检查方式**：扫描正文是否有"修为境界 = 职业等级"的硬性绑定句式。
```

效果：此红线防止模型把项目复杂的独立评价体系简化为"修为决定一切"。

### 示例 D · 补充里程碑的正确入口

项目红线文件中：

```markdown
### 红线 · 策划文档归档入口

**规则**：当用户要求"补充里程碑 / 补充钩子"时，技能**必须**更新 `plots/milestones/` 或 `plots/active-hooks/`，**严禁**往 `contents/` 下的正文塞结构性内容。

**违规表现**：在用户让"补充里程碑"时，回去修改 chapter-*.txt 添加"本五章总结"。

**合规做法**：只写 `plots/milestones/vol-N-ch-Y-summary.md`，正文原封不动。
```

效果：此红线防止策划内容污染正文。

---

## 八、小结

- 技能通过"自动搜索 + 外挂加载"机制支持项目特化
- 项目侧通过一份 `novel-project.md` 即可注入所有特化规则
- 通用红线不可被覆盖，项目红线只能叠加而非放宽
- 找不到红线文件时技能仍可工作，但会提示用户创建
