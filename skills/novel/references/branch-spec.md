# 技能内建分支步骤执行规范

> 本文件定义 `novel` 技能内建的三个分支步骤（分支角色）的执行规范。当技能启动并路由到特定工作流时，技能执行者将自动切换为对应的分支角色，并遵守其定位与硬约束。
> 本规范为单一真相源，执行者必须实时加载并严格遵守。

---

## 一、世界规划分支 (world-builder role)

### 1.1 核心定位
承担 `novel` 技能 `init`（项目初始化）工作流 §1.2 与 `plan`（卷规划）工作流 §2.2 的主导执行：
1. **世界观构建** — 终极规则、核心矛盾、力量体系、地理社会、主题内核。
2. **角色架构** — 主角三要素 + 配角四问 + 反派四问。
3. **冲突架构** — 至少 2 种冲突类型与升级曲线。
4. **卷 Arc 设计** — 4-5 个 Arc，每个 Arc 具备清晰的情绪弧光、高潮点及收尾悬念。
5. **钩子体系** — 长线 / 中线 / 短线三层嵌套。
6. **首批蓝图** — 为目标卷首批 5 章生成符合 8 模块模板的章节蓝图。

### 1.2 执行规范
执行时必须实时加载以下参考规范：
- **金庸写作视角（强制加载）**：`jin-yong-writing-perspective` 技能的 7 个核心心智模型 + 10 个叙事技法（用于人物建模、Arc 设计、冲突推演、钩子体系安排）
- 蓝图模板与完备性校验：`references/blueprint-template.md`
- 章节结构与钩子体系：`references/chapter-structure.md`
- 人物设计规则：`references/character-design.md`（含金庸人物优先原则）
- 冲突设计规则：`references/conflict-design.md`
- 写作风格与文风：`references/writing-style.md`（含金庸古典主义基底）
- 里程碑规则：`references/milestone-rules.md`
- 产物与路径规范：`references/output-spec.md`
- 项目红线追加项：根据 `references/project-override-spec.md` 发现的项目红线文件

### 1.3 工作流程
- **项目初始化 (init)**：
  - **新项目模式**：与用户对话明确类型、读者、爽点及基调；构建核心世界观、核心角色三要素、冲突架构与三层钩子体系。
  - **增量确认模式**：读取既有设定，列出要点供用户确认不覆盖，仅补齐用户指定部分，不擅改既有设定。
- **卷规划 (plan)**：
  - 读取总大纲、上卷末里程碑与所有活跃钩子；通读 `人物体系/` 下所有角色档案（掌握背景、性格、动机、言行模式，确保人物动态无 OOC）。
  - 生成 `outlines/volume-{N}-outline.md`。
  - 为首批 5 章生成 8 模块蓝图，并提交进行完备性校验。

### 1.4 硬约束（不允许的行为）
- 不得只生成卷大纲而不生成章节蓝图。
- 蓝图不得少于 8 模块，严禁自创蓝图模板。
- 未跑通蓝图完备性校验（B1-B8）前不得交付蓝图。
- 不得直接覆盖已存在的蓝图（应提示用户更新或保留）。
- 不得在未通读相关人物体系文档的情况下规划大纲和蓝图，大纲中的人物动态不得与人物设定相悖（OOC）。

## 二、章节创作分支 (story-engine role)

### 2.1 核心定位
承担 `novel` 技能 `write`（章节创作）工作流 §3.2/§3.3、`milestone`（里程碑）工作流 §4.2 和 `hook`（钩子）工作流 §5.2 的主导执行：
1. **蓝图补全** — 在 `write` 前置检查中，若发现缺少章节蓝图，自动补生成并进行完备性校验。
2. **正文创作** — 严格依据蓝图和文风规则，写出章节正文。
3. **里程碑产出** — 在 5/10/15/20 等 5 章节点，自动生成包含 A 节（阶段成果）和 B 节（下一阶段入口）的完整里程碑。
4. **钩子维护** — 在正文涉及钩子引入、推进或回收时，同步更新对应的钩子追踪文档。
5. **接受回修** — 接收质检分支反馈的 FAIL 清单，在 2 轮上限内对正文进行修订并重交。

### 2.2 执行规范
执行时必须实时加载以下参考规范：
- **金庸写作视角（强制加载）**：`jin-yong-writing-perspective` 技能的 7 个核心心智模型 + 10 个叙事技法（用于人物刻画、情感表达、场景呈现、对话设计、开篇收尾）
- 蓝图模板与校验：`references/blueprint-template.md`
- 章节结构：`references/chapter-structure.md`
- 写作风格与文风配比：`references/writing-style.md`（含金庸古典主义基底与间接呈现技法）
- 去 AI 痕迹：`references/anti-ai-patterns.md`
- 打斗场面规范（M7 战斗时）：`references/combat-scene.md`
- 人物设计规则：`references/character-design.md`（含金庸人物优先原则与道德灰度）
- 冲突设计规则：`references/conflict-design.md`
- 里程碑双节规则：`references/milestone-rules.md`
- 创作节奏与修玄用词：`references/writing-conventions.md`
- 产物与路径规范：`references/output-spec.md`
- 项目红线追加项：项目根或子目录的项目红线文件

### 2.3 工作流程
- **创作前准备**：读取最新里程碑、活跃钩子、卷大纲、章节蓝图、涉及的出场角色档案及引用的体系文档。
- **章节创作**：按 M6 结构创作正文，落地为 `contents/volume-{N}/chapter-{X}.txt`。
- **接受回修**：收到质检反馈的 FAIL 指令后，逐项对正文及上下文进行逻辑连带修改，重新提交。

### 2.4 硬约束（不允许的行为）
- 不得跳过前置完备性校验直接创作正文。
- 不得出产少于 8 模块的蓝图。
- 5 章节点的里程碑必须包含 A+B 双节，缺 B 节（下一阶段入口）直接判定为红线 FAIL。
- 不得在不引入新钩子的情况下将 B 节写为空壳。
- 严禁在正文中夹带任何策划/结构性内容（如“本章总结”）。
- 在 `milestone` 或 `hook` 工作流中，只修改对应策划文档，不得触碰正文 `contents/`。
- 未通读出场角色档案前不得写正文；严禁在正文中写出任何 OOC（人物言行不符设定）内容。

## 三、质量质检分支 (quality-editor role)

### 3.1 核心定位
承担 `novel` 技能 `write`（章节质检）工作流 §3.4 和 `check`（质量回归）工作流 §6.2 的质检与回修裁决执行：
1. **红线检查** — 严格执行通用红线（R1-R9）、蓝图 M7 触发的专项红线（T1-T4）及项目红线追加项。
2. **黄线检查** — 扫描非阻塞性质量改进项（Y1-Y8）。
3. **回修裁决** — 判定质检结果为 PASS 或 FAIL，向创作分支下达回修指令，最多回修 2 轮。
4. **质检报告** — 产出符合规范的结构化质检报告。

### 3.2 执行规范
执行时必须实时加载以下参考规范：
- **金庸写作视角（加载检查）**：`jin-yong-writing-perspective` 技能的模型1（人物驱动情节）、模型2（情感具象化）、模型5（道德灰度）——质检时检查正文是否遵循了金庸式人物/情感/灰度原则
- 红黄线检查清单与报告模板：`references/quality-checklist.md`
- 写作风格与文风判定：`references/writing-style.md`
- 去 AI 痕迹与禁用词：`references/anti-ai-patterns.md`
- 章节结构与段落字数：`references/chapter-structure.md` 及 `references/writing-conventions.md`
- 里程碑规则：`references/milestone-rules.md`
- 项目红线发现与加载协议：`references/project-override-spec.md`

### 3.3 工作流程
- **章节质检流程 (write §3.4)**：
  1. 载入通用红线及项目红线追加项。
  2. 扫描蓝图 M7 触发标记（战斗/突破/情感/情绪），叠加专项红线。
  3. 逐项检查，判定结果：全部 PASS 则运行黄线检查，生成报告并交付；任一 FAIL 则触发回修，并将 FAIL 详情（位置、问题、修正建议）返回。
- **回修裁决规则**：
  - 第 1 轮 FAIL -> 返回清单要求回修 -> 重交后跑**完整红线清单**。
  - 第 2 轮 FAIL -> 再次回修 -> 重交后跑完整清单。
  - 第 3 轮仍 FAIL -> 停止自动回修，生成报告，提示用户人工介入。
- **质量回归流程 (check §6.2)**：
  - 针对指定范围的所有章节正文与蓝图，只运行质检生成单章及汇总报告，**不自动改写正文**。

### 3.4 硬约束（不允许的行为）
- 不得跳过蓝图 M7 触发标记扫描，不得遗漏任何专项红线。
- 不得在红线 FAIL 时将章节标记为“已交付”。
- 每轮重新质检必须跑完整清单，不得只跑上轮的 FAIL 项。
- 不得擅自把红线降级为黄线。
- 不得凭文学直觉放行章节，红线判定必须有清单条款依据。
- 在 `check` 回归中只出报告，禁止自动修改正文。
