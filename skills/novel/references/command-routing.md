# 自然语言路由协议

> 本文件定义如何把用户的**自然语言**请求自动路由到对应的工作流，自动定位当前卷/章，**无需用户输入任何参数**。
> 详细工作流定义见 [../novel.md](../novel.md)。

---

## 一、核心原则

1. **零参数**：用户**不需要**输入 `/novel write <vol> <chapter>` 这种命令，技能必须能从自然语言推断参数
2. **自动定位**：卷号、章号、钩子名等一切参数，由技能扫描项目目录结构自动计算
3. **路由透明**：技能执行前必须在回复开头**声明**识别到的意图与定位结果，让用户知晓路由结果，但不要求用户反向确认
4. **意图不明 → 给选项**：如果用户表达可匹配多个工作流，技能必须给出 2-3 个明确选项让用户选择，**禁止**擅自推断

---

## 二、工作流总表

技能支持 6 个工作流：

| 工作流 ID   | 主导分支步骤                                   | 主要产物                                 | 触发语言示例                     |
| ----------- | ---------------------------------------------- | ---------------------------------------- | -------------------------------- |
| `init`      | 世界规划分支 (world-builder)                   | 世界观总纲、体系文档                     | "初始化""新建小说""构建世界观"   |
| `plan`      | 世界规划分支 (world-builder)                   | 卷大纲 + 该卷全部蓝图                    | "规划卷 X""做卷大纲""安排下一卷" |
| `write`     | 创作与质检分支 (story-engine + quality-editor) | 章节蓝图（若缺）+ 正文 + 质检报告        | "写下一章""继续写""第 X 章正文"  |
| `milestone` | 章节创作分支 (story-engine)                    | 里程碑文档（仅 `plots/milestones/`）     | "补充里程碑""写里程碑""五章总结" |
| `hook`      | 章节创作分支 (story-engine)                    | 钩子追踪文档（仅 `plots/active-hooks/`） | "补充钩子""新增钩子""更新钩子 X" |
| `check`     | 质量质检分支 (quality-editor)                  | 质量回归报告（只读不改）                 | "检查""校验""质检""回归"         |

## 三、路由映射表（自然语言 → 工作流）

### 3.1 章节创作（write）

**触发词**（全匹配或近似匹配）：

- 写下一章 / 继续写 / 写新章节
- 写第 X 章 / 第 X 章正文 / 来第 X 章
- 下一章写完它 / 给我来个第 X 章
- 正文 / 写正文 / 章节正文

**参数自动定位**：

1. 扫描 `contents/volume-*/chapter-*.txt`，找到所有已完成章节
2. 识别**最新已完成章节**（例如 `vol-1-ch-12`）
3. **目标章节 = 最新 + 1**（例如 vol-1-ch-13）
4. 若用户指定了具体章号（如"写第 15 章"），则**尊重用户指定**，不自动+1

**路由声明示例**：

> ✅ 已识别意图：**章节创作**
> 📍 自动定位：**卷 1 · 第 13 章**（基于已完成的最新章节 ch-12 + 1）
> 🔄 将启动：章节创作工作流（前置检查 → 蓝图校验 → 正文 → 质检 → 钩子/里程碑更新）

### 3.2 里程碑（milestone）

**触发词**：

- 补充里程碑 / 写里程碑 / 做里程碑
- 五章总结 / 阶段总结
- 补全 vol-X-ch-Y 里程碑

**参数自动定位**：

1. 扫描 `contents/` 找到最新已完成章节 `vol-{N}-ch-{Y}`
2. 计算最近一个**未生成或未合格**的 5 章节点（即 Y 向下取 5 的倍数）
3. 若该节点对应里程碑**不存在**：走"新建里程碑"
4. 若存在但**缺 B 节**（下一阶段入口）：走"补全里程碑"

**关键规则（红线）**：

- 只读写 `plots/milestones/`
- **严禁**触碰 `contents/` 下的正文
- 生成后必须同时含 A 节 + B 节（见 [milestone-rules.md](milestone-rules.md)）

**路由声明示例**：

> ✅ 已识别意图：**补充里程碑**
> 📍 自动定位：**卷 1 · 第 10 章里程碑**（检测到 ch-10 里程碑缺少 B 节）
> 🔄 将启动：里程碑工作流（只改 `plots/milestones/`，不动正文）

### 3.3 钩子（hook）

**触发词**：

- 补充钩子 / 新增钩子 / 建钩子
- 更新钩子 {名称}
- 钩子 X 推进了

**参数自动定位**：

- 若用户指定了钩子名：按名称匹配 `plots/active-hooks/hook-{名称}.md`
- 若未指定：列出所有活跃钩子让用户选择（给选项，不自动推断）

**关键规则（红线）**：

- 只读写 `plots/active-hooks/`
- **严禁**触碰 `contents/` 下的正文
- **严禁**触碰 `plots/milestones/`（钩子和里程碑分工明确）

**路由声明示例**：

> ✅ 已识别意图：**更新钩子**
> 📍 钩子名：**母亲遗物**
> 🔄 将启动：钩子工作流（只改 `plots/active-hooks/hook-母亲遗物.md`）

### 3.4 卷规划（plan）

**触发词**：

- 规划卷 X / 卷 X 大纲 / 做卷大纲
- 安排下一卷 / 下一卷规划
- 卷 X 章节蓝图

**参数自动定位**：

1. 扫描 `outlines/volume-*-outline.md`，找到最新已完成的卷 N
2. **目标卷 = N + 1**
3. 若用户明确指定（"规划卷 3"），尊重用户指定

**关键规则**：

- 产出 `outlines/volume-{N}-outline.md` + 该卷**全部**章节蓝图
- 生成完蓝图立即跑**蓝图完备性校验**（见 [blueprint-template.md](blueprint-template.md)）
- 不合格蓝图阻塞流程

**路由声明示例**：

> ✅ 已识别意图：**卷规划**
> 📍 自动定位：**卷 2**（基于已完成卷 1）
> 🔄 将启动：卷规划工作流（读总大纲 → 读上卷末尾里程碑 → 生成卷 2 大纲 + 全部蓝图 → 蓝图完备性校验）

### 3.5 质量回归（check）

**触发词**：

- 检查 / 校验 / 质检 / 回归
- 检查卷 X / 检查 vol-1
- 跑一下红线 / 对存量章节做检查

**参数自动定位**：

- 若未指定范围：默认检查**当前卷全部已完成章节**
- 若指定卷：检查该卷全部章节
- 若指定章节范围（如 ch-1 到 ch-10）：按范围检查

**关键规则**：

- **只出报告**，不自动修改正文
- 报告路径：`quality-control/correction-suggestions/vol-{N}-ch-{A}-{B}-quality-report.md`
- 由用户决定是否根据报告触发回修

**路由声明示例**：

> ✅ 已识别意图：**质量回归**
> 📍 检查范围：**卷 1 · 第 1-12 章**（当前卷已完成章节）
> 🔄 将启动：质量回归工作流（逐章跑红线/黄线 → 只出报告不改正文）

### 3.6 项目初始化（init）

**触发词**：

- 初始化 / 新建小说 / 新建项目
- 构建世界观 / 搭建世界
- 开始一本新书

**参数自动定位**：

- 检查当前项目根是否已有 `核心世界观总纲.md` 或同类文档
- 若**已存在** → 进入"增量确认"模式，不得覆盖
- 若**不存在** → 按世界规划分支 (world-builder) 流程依次产出

**路由声明示例（存量项目）**：

> ✅ 已识别意图：**项目初始化**
> ⚠️ 检测到已存在世界观总纲（`核心世界观总纲.md`），将进入**增量确认**模式，不覆盖既有设定
> 🔄 将启动：初始化工作流（只补齐缺失的体系文档）

## 四、意图不明的处理

当用户的表达可以匹配 2 个及以上工作流，或参数无法自动推断时，**禁止擅自推断**，必须给用户 2-3 个选项：

### 示例 A · 意图不明

**用户**：

> 帮我整理一下

**技能**（不执行任何操作，只给选项）：

> 你的请求我可以匹配到多个工作流，请选择：
>
> | 选项 | 说明                                             |
> | ---- | ------------------------------------------------ |
> | A    | 跑一次**质量回归**，对当前卷已完成章节出质检报告 |
> | B    | 补充缺失的**里程碑**和活跃钩子（只改策划文档）   |
> | C    | 其他意图请告诉我具体想做什么                     |

### 示例 B · 参数不全

**用户**：

> 更新钩子

**技能**（列出活跃钩子让用户选择）：

> 请指定要更新哪个钩子：
>
> | 选项 | 钩子名       | 当前状态                |
> | ---- | ------------ | ----------------------- |
> | 1    | 母亲遗物     | 推进中（ch-8 最后推进） |
> | 2    | 宗门旧事     | 待回收                  |
> | 3    | 新增一个钩子 | -                       |

## 五、路由优先级与冲突解决

### 5.1 当用户同时说了多件事

例如："写下一章，再补充里程碑"：

- 技能必须**按顺序**处理：先 write 工作流（产生正文），write 工作流的后置产物本身就包含里程碑（若为 5 章节点）
- 避免并行执行，避免重复产物

### 5.2 当表达含参数明确信号时

| 用户表达          | 路由  | 自动定位逻辑                                   |
| ----------------- | ----- | ---------------------------------------------- |
| 写下一章          | write | 扫描 contents → 最新 + 1                       |
| 写第 13 章        | write | 明确指定 13                                    |
| 跳过 11 直接写 13 | write | 明确指定 13，但应提示用户 ch-11/12 缺失        |
| 写第 3 卷的开头   | write | 定位到卷 3 第 1 章                             |
| 重写第 5 章       | write | 明确指定 5，且进入"覆盖模式"（需用户二次确认） |

### 5.3 二次确认场景

以下情况即使路由明确，也需要**二次确认**：

- **覆盖已有正文**：用户说"重写"或指定了已存在的章节
- **跨越章节跳写**：例如已完成到 ch-12，用户要求"写 ch-20"
- **删除 / 回滚**：用户要求删除已有产物（始终需要确认）

## 六、路由决策伪代码

```python
def route(user_input):
    # 1. 载入项目红线（决定根目录、题材等）
    project = load_project_overrides()

    # 2. 分类用户意图
    intent = classify_intent(user_input)  # write / milestone / hook / plan / check / init / ambiguous

    if intent == "ambiguous":
        return ask_user_with_options()  # 给 2-3 个选项

    # 3. 提取或推断参数
    params = extract_params(user_input)
    if not params.chapter and intent == "write":
        params.chapter = scan_latest_chapter(project.contents_dir) + 1
    if not params.volume:
        params.volume = infer_volume(project)

    # 4. 二次确认场景
    if is_overwrite_scenario(intent, params):
        return confirm_with_user(params)

    # 5. 声明路由结果
    announce_routing(intent, params)

    # 6. 启动工作流
    run_workflow(intent, params)
```

## 七、实践建议（给技能实现者）

- **宁可停下来问，不要错。** 路由错误的代价（重写章节、污染策划文档）远大于多问一句的代价。
- **始终在回复开头声明路由结果**，让用户有机会在 0 成本时纠正。
- **对"覆盖已有文件"和"跨章节跳写"始终二次确认**，这是技能被用户信任的底线。
- 当新增一类用户表达时，**更新本文件的触发词表**，而不是在代码中硬编码。
