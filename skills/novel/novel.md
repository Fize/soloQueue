# Novel Skill · 主流程

> 本文件是 `novel` 技能的**主流程文档**。所有"做什么、按什么顺序、谁来做、做完产出什么"的执行约束都在这里。
> 具体规则细节全部在 `references/` 下，本文件只做流程编排与强制闸门定义。
> 技能入口见 [SKILL.md](SKILL.md)。

---

## 0 · 启动协议（每次进入任意工作流前必须执行）

### 0.1 载入金庸写作视角

**所有创作类工作流（init/plan/write/milestone/hook）执行前，必须先加载 `jin-yong-writing-perspective` 技能**，将其 7 个核心心智模型和 10 个叙事技法作为创作方法论底层。具体绑定：

- **人物建模** → 金庸模型1（人物驱动情节）+ 模型5（道德灰度）+ 模型6（间接呈现）
- **大纲/Arc设计** → 金庸模型4（隐藏的根）+ 模型7（武侠即人生）+ 技法5（先有结尾）
- **剧情推演** → 金庸模型2（情感具象化）+ 技法1（草蛇灰线）+ 技法2（横云断山）
- **蓝图设计** → 金庸技法7（章回钩子）+ 技法3（忙中作闲）+ 技法9（人物对话即性格名片）
- **正文创作** → 金庸模型6（间接呈现）+ 技法10（闲笔蓄势）+ 技法4（一击两鸣）

**不在创作类工作流（如 check）则跳过此步骤。**

### 0.2 载入项目红线

按 [references/project-override-spec.md](references/project-override-spec.md) §二的优先级搜索项目红线文件：

1. `<project-root>/novel-project.md`
2. `<project-root>/项目红线.md`
3. `<project-root>/CODEBUDDY.md`
4. `<project-root>/CLAUDE.md`

**就近原则**：若项目子目录也有红线文件，优先使用子目录。

**找到则载入**：按 §四加载协议，把项目特化规则合入**质量质检分支 (quality-editor)**。
**没找到则降级**：按通用规则运行，首次回复提示用户创建 `novel-project.md`。

按 [references/command-routing.md](references/command-routing.md) 把用户自然语言映射到 6 个工作流之一：

本技能由人工（或主控 Orchestrator）主导故事架构，将纯机械性、易出错的正文撰写与静态质检工作委托给两个轻量化 Agent：

| 工作流    | 主导角色                          | 协同角色                                             |

| hook      | **我们 (创作者/编排者)**          | -                                                    |

## 1 · init · 项目初始化

### 1.1 前置闸门（缺一不可）

| #   | 检查项                                     | 失败时的处理                                       |
| --- | ------------------------------------------ | -------------------------------------------------- |
| P1  | 当前目录是否可写？                         | 提示用户指定项目根                                 |
| P2  | 是否已存在 `核心世界观总纲.md`（或同类）？ | 若存在 → 切换到 **增量确认模式**，不得覆盖既有设定 |
| P3  | 项目红线文件是否存在？                     | 不存在 → 提示用户考虑创建，但不阻塞                |

### 1.2 执行（主导：我们 - 创作者/编排者）

**新项目模式**（P2 失败，即不存在世界观总纲）：

1. 与用户**对话**构建类型与基调（目标读者、核心爽点、情感基调）
2. 构建核心世界观（终极规则、不可调和矛盾、普通人如何生存、独特魅力）
3. **构建核心角色**（主角三要素 + 配角四问 + 反派四问，见 [character-design.md](references/character-design.md)）——**必须调用金庸视角进行人物建模**：
   - **金庸模型1（人物驱动情节）**：人物的欲望/恐惧/矛盾是故事发动机，情节为人物服务
   - **金庸模型5（道德灰度）**：没有纯黑纯白——每正派有皱眉之处，每反派有同情之处
   - **金庸模型6（间接呈现）**：重要人物通过他人视角折射，不直接铺陈其全部特质
   - **金庸技法9（对话即性格名片）**：每人一套独特的说话方式
4. 构建冲突架构（至少 2 种冲突类型，见 [conflict-design.md](references/conflict-design.md)）
5. **构建钩子体系**（长线 / 中线 / 短线三层嵌套）——**必须调用金庸技法1（草蛇灰线）**：埋下"隐藏的根"驱动所有情节，伏笔要千里之外提前埋设。

**增量确认模式**（P2 通过，即已有世界观）：

1. 读取既有 `核心世界观总纲.md` + 所有体系文档
2. **列出**当前体系文档清单、当前设定要点，请用户确认不覆盖
3. 仅补齐**用户指定要补**的部分
4. 不得擅自修改已有体系文档（除非用户明确要求）

### 1.3 后置产物（必须，缺一即不合格）

- `核心世界观总纲.md`（文件名可由项目红线覆盖）
- 各设定与百科文档（如 `世界设定.md`、`力量体系.md`、`地理与势力设定.md` 等，名字可由题材自拟）
- `outlines/总大纲.md`（严格按照 `references/master-outline-template.md` 结构生成）
- 建议同时创建 `novel-project.md` 作为项目红线文件（可选但强烈推荐）

## 2 · plan · 卷规划

### 2.1 前置闸门（缺一不可）

| #      | 检查项                                                               | 失败时的处理                                |
| ------ | -------------------------------------------------------------------- | ------------------------------------------- |
| P1     | 载入项目红线（§0.1）                                                 | -                                           |
| P2     | 是否存在 `outlines/总大纲.md` 或等价文件？                           | 不存在 → 提示先走 init                      |
| P3     | 若目标卷 > 1，上一卷末尾里程碑是否存在？                             | 不存在 → 提示先补里程碑                     |
| P4     | 扫描所有 `plots/active-hooks/` 下活跃钩子                            | 无目录 → 自动创建                           |
| **P5** | **读取 `人物体系/` 下所有角色档案**（主角 + 配角 + 反派 + 势力成员） | **不存在 → 阻塞，人物体系不完备不得规划卷** |
| **P6** | **运行项目完整性静态校验门禁**：在沙箱中运行 `python skills/novel/scripts/novel_lint.py --check-project` | **红线错误数 > 0 → 阻塞**，提示并输出未对齐章节与断档里程碑清单 |

### 2.2 执行（主导：我们 - 创作者/编排者 / 校验：质量质检 `quality-checker` · 规划时绑定金庸视角）

1. **调用金庸视角进行卷规划推演**：
   - **Arc 结构设计** → 金庸模型4（隐藏的根）：找到驱动整卷的"埋藏的秘密或矛盾"。金庸技法5（先有结尾）：先确定本卷的最终落点，再反向推导路径。
   - **人物动态规划** → 金庸模型1（人物驱动情节）：检查每一条人物线——情节走向是否出自人物性格？金庸模型5（道德灰度）：正反派是否有令人挣扎的灰度？
   - **钩子体系安排** → 金庸技法1（草蛇灰线）：伏笔什么时候埋、什么时候收？金庸模型4的"根"决定了钩子的深度——根越深，钩子能拉越远。
   - **冲突升级曲线** → 金庸模型7（武侠即人生）：冲突不只是打斗升级，更是人性困境的层层加深。
2. **通读 `人物体系/` 下所有角色档案**（主角、配角、反派、势力成员等），掌握每个角色的：
   - 背景出身、性格特质、核心动机
   - 行为模式、说话方式、底线与禁忌
   - 与他人的关系网络、立场冲突点
3. 读取总大纲、上卷末尾里程碑、所有活跃钩子
4. 生成 `outlines/volume-{N}-outline.md`，**必须严格按照 `tianyuanjian/outlines/template.md` 的模板格式与 YAML Frontmatter 结构**，并包含以下核心内容：
   - 本卷核心主题（一句话）
   - Arc 结构（4-5 个，每个 Arc 含情绪弧光、高潮点、收尾悬念）
   - 人物动态（各角色目标变化、关系推进、新角色引入时机）——**必须与人物体系设定一致，不得出现 OOC 行为或关系跳跃**
   - 钩子安排（本卷埋设 / 推进 / 回收的钩子）——**必须遵循金庸技法1（草蛇灰线）：伏笔在早期不经意埋下，后期关键处收回**
   - 冲突升级曲线（章与 Arc 层级都要有）
5. 为该卷**首批 5 章**（或至第一个里程碑节点）生成蓝图 `blueprints/volume-{N}/chapter-{X}-blueprint.md`，严格按 [blueprint-template.md](references/blueprint-template.md) 的 9 个模块标准（支持 Frontmatter 级别 `complexity` 复杂度配置）
6. 蓝图完备性校验闸门（由质量质检 `quality-checker` 执行）：
   - 对每份新生成蓝图跑 B1-B9 分级校验（见 [blueprint-template.md](references/blueprint-template.md) §3.2）
   - 任一 FAIL → **阻塞该批次蓝图交付**，返回错误清单要求修订
   - 全部 PASS → 该批次蓝图通过

### 2.3 后置产物（必须）

- `outlines/volume-{N}-outline.md`
- `blueprints/volume-{N}/chapter-{X}-blueprint.md` × 该批次全部章节（首批至少 5 章）
- 所有蓝图标记为"蓝图已审核 · PASS"

### 2.4 不允许的行为

- 不得**只生成大纲不生成蓝图**
- **必须滑窗式生成**：每次卷规划仅允许为当前 5 章周期生成蓝图，**禁止超前一次性生成超过 5 章蓝图文件**（防止剧情微调产生的重构负债）
- 每批次至少生成 5 章蓝图（剩余章节不足 5 章时全部生成）
- 不得跳过完备性校验直接交付
- 不得覆盖已存在的蓝图（应提示用户是要更新还是保留）
- **不得在未通读所有人物体系文档的情况下生成卷大纲或蓝图**
- **卷大纲中的人物动态不得与人物体系设定相悖（OOC），包括但不限于：行为不符性格、动机不符背景、关系进度跳跃、台词不符身份**

## 3 · write · 章节创作

> 这是技能最核心、最长的工作流。**三段强制闸门**（前置 → 执行 → 后置）的每一步都不得绕行。

### 3.1 前置闸门（8 步，任一失败立即停止）

| #      | 检查项                                                                                       | 失败时的处理                                                            |
| ------ | -------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| P1     | 载入项目红线（§0.1）                                                                         | -                                                                       |
| P2     | 读取**最新里程碑**（`plots/milestones/` 下最新一份）                                         | 不存在且非第 1 章 → 阻塞，提示先走 milestone 工作流                     |
| P3     | 扫描 `plots/active-hooks/` 列出需推进/回收的钩子                                             | 无目录 → 自动创建（允许）                                               |
| P4     | 读取 `outlines/volume-{N}-outline.md` 中本章所属 Arc 定位                                    | 不存在 → 阻塞，提示先走 plan 工作流                                     |
| P5     | 读取 `blueprints/volume-{N}/chapter-{X}-blueprint.md`                                        | 不存在 → **阻塞**，提示编排者（我们）先完成该章节蓝图规划并进行校验 |
| P6     | 读取相关体系文档（按蓝图 M8 的体系文档引用清单）                                             | 清单项文件不存在 → 阻塞，提示用户创建                                   |
| **P7** | **通读蓝图 M2 中列出的出场角色档案**，核验本章出场角色的背景、性格、经历、行为模式、说话方式 | **角色档案不存在 → 阻塞**，提示用户先创建                               |
| **P8** | **运行项目完整性静态校验门禁**：在沙箱中运行 `python skills/novel/scripts/novel_lint.py --check-project` | **红线错误数 > 0 → 阻塞**，输出项目断档与对齐失败清单，提示整改 |

**任一前置失败**：立即停止，向用户**明确报告缺失项**与修复建议（例如"未找到 ch-13-blueprint.md，请编排者先补齐蓝图"），**不得假设并继续**。

### 3.2 执行（主导：故事写手 `story-writer` · 强制绑定金庸视角）

前置全 PASS 后，**故事写手必须在 `jin-yong-writing-perspective` 技能指导下工作**：

1. 故事写手 `story-writer` 根据**渐进式按需加载规则（硬约束）**进行上下文加载：
   - 仅允许读取并加载蓝图 **M2（出场角色）** 明确列出的对应 `人物体系/{姓名}.md` 档案；
   - 对于百科与设定，**必须且仅允许使用 `novel_lint.py --read-section` 脚本按需提取** 蓝图 **M8（设定/百科引用）** 中声明的具体锚点（如 `地理与势力设定.md#落霞谷`）下的局部设定片段，严禁全量读取整份大文件；
   - 对于钩子，**必须限制仅读取蓝图 M5 中涉及钩子追踪文件中的 Section 1 (读者已知线索)**，绝对禁止全量加载包含 Section 2 (创作者大纲占位/未披露揭底) 的段落，以实现读者与作者的物理视角隔离，防范 AI “全知视角”剧透；
   - 以及必要的时序上下文（里程碑 / 活跃钩子 / 卷大纲 / 本章蓝图 / 项目红线）。
   - **严禁全量加载整个 `人物体系/` 文件夹或未经脚本抽取的未声明设定百科，以防止注意力稀释与上下文污染。**
2. 调用金庸视角进行创作推演：
   - **人物刻画**：用金庸模型1（人物驱动情节）检验——本章的情节发展是否由人物性格自然推动，而非作者强制安排？用金庸模型5（道德灰度）审视——每个出场人物是否都有值得同情的侧面或让人皱眉的缺点？
   - **情感表达**：用金庸模型2（情感具象化）创作——不告诉读者"很伤心"，而是写动作、写物件、写仪式。情感必须有"代价"。
   - **场景呈现**：用金庸模型6（间接呈现）——重要人物出场/关键转折通过他人反应来折射，不直接铺陈。
   - **结构节奏**：用金庸技法2（横云断山）和技法3（忙中作闲）安排章节内部的张弛节奏。
   - **对话设计**：用金庸技法9（人物对话即性格名片）——不看名字就能认出是谁在说话。
   - **开篇/收尾**：用金庸技法10（闲笔蓄势）开篇蓄积情绪，用技法7（章回钩子）收尾留悬念。
3. 按蓝图 M6 的"凤头 / 猪肚 / 豹尾"结构**创作正文**
4. 正文必须遵守：
   - [writing-style.md](references/writing-style.md)（文风 + 配比）
   - [fusion-brushwork.md](references/fusion-brushwork.md)（融合笔法：沉默见证者、内心独白禁令、场景执行规范）
   - [anti-ai-patterns.md](references/anti-ai-patterns.md)（禁用词 / Show-not-Tell / 五感）
   - [chapter-structure.md](references/chapter-structure.md)（凤头豹尾）
   - [writing-conventions.md](references/writing-conventions.md)（段落上限 8 行、用词修玄化）
   - 项目红线追加的所有约束
5. 正文落地到：`contents/volume-{N}/chapter-{X}.txt`（`.txt` 格式，中文）

### 3.3 后置闸门：红线质检与自动回修（主导：质量质检 `quality-checker`）

**强制闭环**（见 §7 详细说明，本节先给流程）：

```
故事写手 story-writer 产出正文
   ↓
质量质检 quality-checker 强制在沙箱执行静态校验命令：
python skills/novel/scripts/novel_lint.py contents/volume-{N}/chapter-{X}.txt --report-dir quality-control/correction-suggestions/
   ↓
┌─────────────────────────────────┬──────────────────────────────────┐
│ Linter 返回 exit 0 (PASS)       │ Linter 返回 exit 1 (FAIL)        │
├─────────────────────────────────┼──────────────────────────────────┤
│ 1. 跑人工/Subagent 专项红线     │ 1. 解析 Linter 报错及 JSON 文件  │
│ 2. 扫描黄线项                   │ 2. 回修第 1 轮 → 重新运行 Linter  │
│ 3. 补全质检报告，章节交付       │ 3. 再次失败 → 回修第 2 轮        │
│                                 │ 4. 仍失败 → 停止并提示人工介入   │
└─────────────────────────────────┴──────────────────────────────────┘
```

### 3.4 产物归档（红线通过后才做）

- **正文**：`contents/volume-{N}/chapter-{X}.txt`
- **质检报告**：`quality-control/correction-suggestions/vol-{N}-ch-{X}-quality-report.md`
- **钩子更新**：若本章推进/回收钩子 → 更新对应 `plots/active-hooks/hook-{名称}.md`
- **新钩子**：若本章引入跨 5 章以上新钩子 → 新建 `plots/active-hooks/hook-{名称}.md`
- **里程碑**：若本章是 5、10、15、20…… 章 → **必须**生成 `plots/milestones/vol-{N}-ch-{Y}-summary.md`，**同时包含** A 节（阶段成果）+ B 节（下一阶段入口），仅 A 节 = FAIL

### 3.5 write 工作流的不允许行为

- 不得跳过前置任一步骤
- 不得在红线 FAIL 时把章节标记为完成
- 不得在 5 章节点产出"只有 A 节"的里程碑
- 不得在不引入新钩子的情况下产出"空的下一阶段入口"
- 不得往正文里塞结构性内容（如"本章总结""人物状态表"）——这些属于策划文档
- **不得在未通读蓝图 M2 列出的出场角色档案的情况下创作正文**
- **不得写出与人物体系设定相悖的 OOC 内容，包括但不限于：行为不符性格、动机不符背景、对话不符身份、反应不符经历、关系进度跳跃**

## 4 · milestone · 里程碑

### 4.1 前置闸门

| #   | 检查项                                                               | 失败时的处理                                 |
| --- | -------------------------------------------------------------------- | -------------------------------------------- |
| P1  | 载入项目红线                                                         | -                                            |
| P2  | 识别目标里程碑：`vol-{N}-ch-{Y}-summary.md`（Y = 最近一个 5 章节点） | 若 Y 对应章节正文不存在 → 提示用户先写完正文 |
| P3  | 读取 ch-{Y-4} 到 ch-{Y} 的所有正文                                   | 缺章 → 阻塞                                  |
| P4  | 读取所有活跃钩子当前状态                                             | -                                            |

### 4.2 执行（主导：章节创作分支 `story-engine`）

按 [milestone-rules.md](references/milestone-rules.md) §二模板填写里程碑：

- **A 节 · 阶段成果**：人物状态、资产清单、地理时间、四问（解决了什么 / 失去了什么 / 拿到了什么 / 认清了什么）
- **B 节 · 下一阶段入口**：至少明确一项（下一目标 / 下一危机 / 下一悬念 / 下一去处 / 下一抉择）
- **钩子状态快照**：与 `plots/active-hooks/` 同步
- **本五章回顾摘要**：3-5 句话

### 4.3 后置闸门

| #   | 检查项                                               | 失败时的处理                 |
| --- | ---------------------------------------------------- | ---------------------------- |
| Q1  | A 节是否齐全？（人物状态、资产、地理时间、四问都在） | 不齐 → 重写                  |
| Q2  | B 节是否至少明确一项下一阶段入口？                   | 不明确 → **红线 FAIL**，重写 |
| Q3  | 钩子快照是否与 `plots/active-hooks/` 一致？          | 不一致 → 同步                |

### 4.4 milestone 工作流的红线

- **只改 `plots/milestones/`，不得触碰 `contents/` 下正文**
- **必须同时有 A 节 + B 节**，缺 B 节 = 不合格
- 若本章不是 5 章节点但用户仍要求"补里程碑"，应引导用户明确"是要补哪一期里程碑（最近一个 5 章节点）"

## 5 · hook · 钩子

### 5.1 前置闸门

| #      | 检查项                                                         | 失败时的处理                                            |
| ------ | -------------------------------------------------------------- | ------------------------------------------------------- |
| P1     | 载入项目红线                                                   | -                                                       |
| P2     | 识别目标钩子：从用户表达提取名称，或列出活跃钩子让用户选择     | 无法识别 → 给选项让用户选                               |
| P3     | 读取 `plots/active-hooks/hook-{名称}.md`（若是更新）或准备新建 | -                                                       |
| **P4** | **通读 `人物体系/` 下所有角色档案**，特别是钩子涉及的角色      | **未通读 → 阻塞**，不得在不掌握角色设定的情况下安排钩子 |

### 5.2 执行（主导：章节创作分支 `story-engine`）

1. **通读 `人物体系/` 下所有角色档案**，掌握钩子涉及角色的背景、性格、动机、行为模式
2. 按 [chapter-structure.md](references/chapter-structure.md) §2.3 的字段填写钩子追踪：

- **引入章节**
- **核心悬念**
- **线索掉落计划**（每隔几章掉一块新线索）
- **回收节点**
- **当前状态**（未引入 / 已引入 / 推进中 / 待回收 / 已回收）
- **涉及角色**（列出钩子涉及的角色，并确保钩子安排与角色设定一致）

### 5.3 hook 工作流的红线

- **只改 `plots/active-hooks/`，不得触碰 `contents/` 与 `plots/milestones/`**
- 钩子追踪必须同步到最近一次里程碑的"钩子状态快照"
- **钩子安排必须与涉及角色的背景、性格、经历一致**，不得为制造悬念而让角色做出 OOC 行为

## 6 · check · 质量回归

### 6.1 前置闸门

| #   | 检查项                                 | 失败时的处理                          |
| --- | -------------------------------------- | ------------------------------------- |
| P1  | 载入项目红线                           | -                                     |
| P2  | 识别检查范围：默认当前卷全部已完成章节 | 可由用户指定                          |
| P3  | 范围内每章对应蓝图是否存在？           | 缺 → 仍继续跑但在报告中标注"蓝图缺失" |

### 6.2 执行（主导：质量质检分支 `quality-editor`）

1. 逐章加载正文 + 蓝图 + 项目红线
2. 跑红线 R1-R9
3. 按蓝图触发标记加载 T1-T4 专项红线
4. 跑黄线 Y1-Y8
5. 跑项目红线追加项
6. 生成**单章报告**与**汇总报告**

### 6.3 后置产物

- 每章：`quality-control/correction-suggestions/vol-{N}-ch-{X}-quality-report.md`
- 汇总：`quality-control/correction-suggestions/vol-{N}-ch-{A}-{B}-quality-report.md`
- 汇总报告含"必须修订"与"建议修订"两张清单

### 6.4 check 工作流的红线

- **只出报告，不自动修改正文**
- 由用户决定是否根据报告触发回修（触发方式：用 write 工作流的差值回修模式）

## 7 · 红线质检与自动回修闭环（详细说明）

> 本节是 write 工作流 §3.4 的详细展开，是整个技能最核心的强制闭环。

### 7.1 执行流程

```
              [创作分支产出正文]
                       │
                       ▼
[Step 1] 强制在沙箱执行静态校验命令：
         python skills/novel/scripts/novel_lint.py contents/volume-{N}/chapter-{X}.txt --report-dir quality-control/correction-suggestions/
                       │
         ┌─────────────┴─────────────┐
         ▼                           ▼
  [Linter 返回 exit 1]       [Linter 返回 exit 0 (PASS)]
         │                           │
         │                           ▼
         │                  [Step 2] 加载项目红线追加项
         │                           │
         │                           ▼
         │                  [Step 3] 读取蓝图 M7 · 触发标记并加载 T1-T4 专项
         │                           │
         │                           ▼
         │                  [Step 4] 跑人工/Subagent 专项与黄线 Y1-Y8
         │                           │
         ▼                           ▼
[Step 5] 结果判定 ───────────────────┘
  ┌────────────────────────────┬────────────────────────┐
  │ 全部 PASS                  │ 任何一步 FAIL          │
  │ → 补全 Markdown 质检报告    │ → 触发回修             │
  │ → 标记章节为"已交付"        │                        │
  └────────────────────────────┴────────────────────────┘
```

### 7.2 回修规则与精准差值补丁 (Diff-based Self-Healing)

当静态 Linter 或人工红线拦截章节正文后，为避免大面积重写导致“修复 A 引入 B”的无限循环和 Token 浪费，**强制采用差值式回修模式 (Diff-based Patch)**：

| 轮次 | 行为 | 修复动作 |
| --- | --- | --- |
| **第 1 轮** FAIL | 回修分支 `story-engine` 精确对应行号进行差值合并 | 1. 自动将 Linter 报错的 JSON 错误列表与对应行号上下文传给 `story-engine`。<br>2. 限制 Agent 仅针对出错的行号范围输出 `<<<< StartLine-EndLine ... ==== ... >>>>` 格式的替换块。<br>3. 自动执行 `python skills/novel/scripts/apply_patch.py` 脚本合并修改。 |
| **第 2 轮** FAIL | 再次进行精准差值补丁修复 | 同上，附带最新 Linter 的报错详情与行号。 |
| **第 3 轮** FAIL | **停止自动回修**，生成报告，提示用户人工介入 | - |

**硬性约束**：
1. **禁止全篇重写**：回修提示词必须限制 Agent 不得输出整章正文，必须以指定格式输出修改块。
2. **闭环全量回归**：每次合并 Patch 后，**必须**重新运行完整的 `novel_lint.py` 进行静态闭环扫描，以验证合并效果，防止 Patch 合并时产生新的语法或红线错误。

### 7.3 质检报告格式

见 [quality-checklist.md](references/quality-checklist.md) §五。报告必须含：

- 总分（X/10）
- 红线结果（PASS / FAIL · 第 N 轮）
- 黄线结果（Y 项需注意）
- 红线项清单（每项：位置 / 问题 / 修正建议）
- 触发标记专项清单
- 黄线项清单
- 项目红线追加项清单
- 结论（是否可交付 / 是否需要人工介入）

### 7.4 不允许的行为

- 不得跳过 Step 3 的触发标记扫描
- 不得在红线 FAIL 时把章节标记为完成
- 不得在某一轮修订中只读 FAIL 项而不重新跑完整清单（避免修一个问题引入另一个问题）
- 不得擅自把红线降为黄线
- 不得绕过质量质检分支 `quality-editor` 直接交付（即使模型自己认为"写得很好"）

### 7.5 双 Agent 的自愈闭环

由于技能已全面整合故事写手与质量质检两大角色，执行时将直接在同一会话中无缝完成角色状态切换（由 `story-writer` 创作过渡到 `quality-checker` 质检与回修裁决），不再具有外部 Agent 缺失的降级风险。
